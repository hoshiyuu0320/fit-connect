# Task 2.3 設計書: 画像からのカロリー推定

**作成日**: 2026-05-09
**対象**: フェーズ2 タスク 2.3（Mobile + Supabase Edge Function）
**前提タスク**: 2.1（Edge Function 基盤、完了）/ 2.2（テキスト入力からのカロリー推定、完了）

## 1. 目的とスコープ

食事写真を撮影/選択して AI（Claude Vision）でカロリー・PFC を推定し、トレーナーに送信するメッセージに自動で `meal_estimation` メタデータを付与する。

### スコープ
- 既存の `MealTagForm`（食事を記録シート）に画像 + AI 推定フローを統合する
- Supabase Edge Function `estimate-meal-nutrition` に Vision 対応を追加
- 確認シートで PFC 数値の編集を可能にし、画像はロックする
- Pro プラン契約者のみ AI 推定を実行（既存の課金ゲートを流用）

### スコープ外（次タスク以降）
- スクリーンショット解析（タスク 2.5）
- 推定履歴・精度フィードバック機能（タスク 2.4 任意項目）
- AI 推定専用バケット作成、cleanup ジョブ（運用観測後に必要なら別タスク）

## 2. ユーザーフロー

```
[食事を記録シート（input phase）]
  ├─ 食事タイプ選択（朝/昼/夕/間食）
  ├─ テキスト入力（任意）
  ├─ 画像選択（最大 3 枚、既存 image_picker そのまま）
  └─ [挿入] タップ
       │
       ▼
[loading phase]
  ├─ 画像があれば: StorageService.uploadAiImage で `${userId}/ai/${uuid}.jpg` に upload
  ├─ Edge Function 呼び出し（image_urls + content + meal_type）
  └─ キャンセルボタン表示（既存通り）
       │
       ├──[成功]──▶ confirm phase
       └──[失敗]──▶ snackbar +「AIなしで送信」アクション
       │
       ▼
[confirm phase]
  ├─ 食品リスト表示（読み取り専用）
  ├─ 画像サムネイル表示（読み取り専用、削除ボタン無し）
  ├─ 合計 kcal/P/F/C 編集可能
  ├─ [戻る] → input phase に復帰（画像 URL は保持、再 upload 防止）
  └─ [送信] → onSendWithEstimation へ
       │
       ▼
[ChatInput がメッセージ送信]
  ├─ 画像 URL は upload 済みのものを再利用（_uploadImagesIfAny は skip）
  └─ messages.metadata.meal_estimation = {calories, P, F, C, foods, source: 'vision'}
```

### サブスクリプション分岐
- **Pro プラン**: 上記フル AI フロー
- **Free プラン**: 画像があっても AI は呼ばず、既存通り「挿入」で text composer に挿入のみ（2.2 と同じ挙動）

## 3. 主要な設計判断

| 判断項目 | 採用案 | 理由 |
|---|---|---|
| 画像送信方式 | Storage upload → public URL を Claude に渡す | `message-photos` バケットが既に public、再利用可能、Edge Function ロジックがシンプル |
| 複数画像の扱い | 全画像を 1 リクエストにまとめて Vision に渡し、合計値を返す | 1 食を複数アングル/別皿で撮るユースケースに自然、API コスト最小 |
| テキスト併用 | 画像 + 任意テキスト（補足コンテキストとして Claude に渡す） | 「ご飯大盛り」等の補足を反映可能、UI を変えずに済む |
| 使用モデル | 動的切替（image あり → Sonnet 4.6 / image なし → Haiku 4.5） | Vision 精度を確保しつつテキスト推定のコストは据え置き |
| Rate limit | 既存 50/client/day と合算（Vision 1 回 = 1 カウント） | 実用上 50 回/日に到達しない、UX シンプル |
| Orphan 画像 | 許容（cleanup は将来課題） | Pro プランのみ + 1 食数百 KB 規模、短期的に問題化しない |
| Orphan 識別 | パスプレフィックス `${userId}/ai/${uuid}.jpg` で分離 | 将来 cleanup スキャン範囲を限定可能、実装コストほぼゼロ |
| Confirm phase の画像編集 | 不可（読み取り専用） | URL の整合性を保つため。変更は「戻る」で input phase へ |

## 4. コンポーネント設計

### 4.1 Supabase Edge Function `estimate-meal-nutrition` の拡張

**変更点**:
- リクエストボディに `image_urls?: string[]` を正式サポート（現状は無視されている）
- 画像が含まれる場合は **Sonnet 4.6**（model ID: `claude-sonnet-4-6`）に動的切替。テキストのみは既存 Haiku 4.5（model ID: `claude-haiku-4-5`）のまま
- Claude API messages に Vision content block を追加：
  ```typescript
  content: [
    ...image_urls.map(url => ({ type: 'image', source: { type: 'url', url } })),
    { type: 'text', text: `食事タイプ: ${meal_type}\n補足: ${content || '(なし)'}` }
  ]
  ```
- system prompt に Vision 対応の指示を追記（次節）。prompt caching の `cache_control` は維持
- レスポンススキーマ・JSON 抽出ロジック（`extractJson`）は既存をそのまま流用
- Rate limit ロジック（`ai_estimation_logs`）は既存をそのまま流用（合算 50/day）
- ログ：`ai_estimation_logs` に既存通り記録（`image_count` カラム追加は任意、最小スコープでは省略）

**System prompt 追記内容**:
- 「複数画像が来たら同一食事の異なる視点／品とみなして合計値を返す」
- 「食事と無関係な画像なら `foods=[]` を返す」
- 「テキストの補足は画像から得た情報の修正に優先する（例：『ご飯大盛り』なら米飯の量を増やす）」

### 4.2 Mobile: `StorageService.uploadAiImage`（新規）

```dart
// services/storage_service.dart
static Future<String?> uploadAiImage(File file, String userId) async {
  final fileName = '${_uuid.v4()}.jpg';
  final filePath = '$userId/ai/$fileName';   // ← prefix で分離
  await SupabaseService.client.storage
      .from(bucketName)
      .upload(filePath, file);
  return SupabaseService.client.storage
      .from(bucketName)
      .getPublicUrl(filePath);
}

static Future<List<String>> uploadAiImages(List<File> files, String userId) async {
  // 並列実行で複数枚対応（既存 uploadImages の逐次実行はそのまま）
}
```

既存 `uploadImage` には触らない。

### 4.3 Mobile: `MealEstimationApi` の拡張

```dart
// features/meal_records/data/meal_estimation_api.dart
Future<MealEstimationResult> estimate({
  required MealType mealType,
  required String content,
  List<String> imageUrls = const [],   // ← 追加
}) {
  return SupabaseService.client.functions.invoke(
    'estimate-meal-nutrition',
    body: {
      'meal_type': mealType.name,
      'content': content,
      if (imageUrls.isNotEmpty) 'image_urls': imageUrls,
    },
  );
}
```

エラーマッピング（`MealEstimationErrorCode`）は既存を流用。

### 4.4 Mobile: `MealTagForm` のロジック変更

`_MealFormPhase.input → loading → confirm` の遷移内に画像 upload を組み込む：

```
挿入ボタンタップ:
  1. setState(_phase = loading)
  2. if (selectedImages.isNotEmpty && uploadedImageUrls.isEmpty) {
       uploadedImageUrls = await uploadAiImages(selectedImages, userId);
     }
     // 「戻る」→「挿入」再試行時は uploadedImageUrls が既に埋まっているので skip
  3. result = await estimate(content, mealType, imageUrls: uploadedImageUrls)
  4. setState(_phase = confirm, _result = result)

戻るボタン（confirm phase）:
  setState(_phase = input)   // uploadedImageUrls はそのまま保持

確認シート → 送信:
  onSendWithEstimation(composedText, result, preUploadedUrls: uploadedImageUrls)
```

**Confirm phase での画像表示**: `MealEstimationConfirmView` に画像サムネイル表示を追加（読み取り専用、削除ボタン無し）。

**「戻る」→ 画像変更時の挙動**:
- ユーザーが画像を削除 → 削除された File に対応する URL は `uploadedImageUrls` から除外（Storage の object はそのまま orphan に）
- ユーザーが画像を追加 → 「挿入」時に未 upload の File のみ新規 upload

**File ↔ URL マッピング**: 順序ベースではなく `Map<File, String>`（`fileToUrlMap`）で保持し、File インスタンス単位で upload 済みかを判定する。`MealTagForm` の State にこの Map を持たせる。

### 4.5 Mobile: `ChatInput` の改修

`onSendWithEstimation` で `preUploadedUrls` を受け取れるようにし、内部で持っている `_selectedImages` を URL に置き換え（再アップロード回避）：

```dart
// chat_input.dart の onSendWithEstimation 経路
Future<void> _sendWithEstimation(
  String text,
  MealEstimationResult estimation,
  List<String> preUploadedUrls,
) async {
  // 通常の _uploadImagesIfAny() は呼ばない
  await sendMessage(
    content: text,
    imageUrls: preUploadedUrls,
    metadata: { 'meal_estimation': estimation.toJson() },
  );
  setState(() => _selectedImages.clear());
}
```

### 4.6 messages.metadata の拡張

既存:
```json
{
  "meal_estimation": {
    "calories": 600,
    "protein_g": 25,
    "fat_g": 20,
    "carbs_g": 75,
    "foods": [...]
  }
}
```

追加（観測用、`parse-message-tags` の挙動には影響しない）:
```json
{
  "meal_estimation": {
    "calories": 600,
    "protein_g": 25,
    "fat_g": 20,
    "carbs_g": 75,
    "foods": [...],
    "source": "vision" | "text"
  }
}
```

`parse-message-tags` webhook の改修は不要（既存ロジックがそのまま動く）。`source` は将来の精度フィードバック用。

## 5. エラーハンドリング・エッジケース

| シナリオ | 挙動 |
|---|---|
| 画像 upload 失敗（一部） | 失敗した画像は除外、残りで AI 推定を続行 |
| 画像 upload 失敗（全て） | snackbar「画像のアップロードに失敗しました」+ input phase に戻す |
| Edge Function: `FORBIDDEN`（free プラン） | `aiFeaturesEnabledProvider` で先に弾くため通常到達しない。到達したら snackbar +「AIなしで送信」 |
| Edge Function: `RATE_LIMIT` | snackbar「本日の AI 推定上限に達しました」+「AIなしで送信」 |
| Edge Function: `ESTIMATION_FAILED` | snackbar「推定に失敗しました」+「AIなしで送信」 |
| Vision が食事を認識できず `foods=[]` | confirm phase に進まず、snackbar「画像から食事を識別できませんでした。テキストで補足して再試行してください」を表示。snackbar action は「AIなしで送信」の 1 つのみ（既存パターン踏襲）。テキスト補足の促しは snackbar 本文で行う |
| ネットワーク・タイムアウト | snackbar +「AIなしで送信」 |
| 「戻る」→ 画像追加 →「挿入」 | 既存 upload 済み URL は保持、新規画像のみ追加 upload |
| 「戻る」→ 画像削除 →「挿入」 | 削除された File に対応する URL は state から除外（Storage object は orphan に） |

### タイムアウト
Vision は応答時間が長い（5〜15 秒想定）。既存のテキスト推定タイムアウトを **30 秒** に延長（テキスト時も同じ値で OK、上回ることはまず無いため分岐不要）。
